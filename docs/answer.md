**Q1. 데이터베이스는 엑셀과 무엇이 다르며, 왜 테이블을 나누어 저장하나요?**
- 차이점: 단순히 많은 데이터를 저장하는 크기의 차이가 아닙니다. 데이터베이스는 테이블 사이의 '관계'를 표현할 수 있으며, 데이터 무결성 제약조건을 통해 잘못된 데이터가 들어오는 것을 원천 차단합니다.

- 나누어 저장하는 이유: 하나의 거대한 테이블에 모든 정보를 넣으면 중복 데이터가 많이 발생하고, 수정 시 데이터 불일치(이상 현상)가 일어납니다. 따라서 역할을 기준으로 테이블을 분리하여 데이터 구조를 단순화하고 정합성을 유지합니다.

**Q2. PK(기본키)와 FK(외래키)의 역할은 무엇이며, 1:N 관계는 데이터를 어떻게 연결하나요?**
- PK (Primary Key): 테이블 내의 모든 행을 고유하게 식별할 수 있는 기준점이 되는 키입니다. 중복될 수 없으며, NOT NULL이어야 합니다.

- FK (Foreign Key): 다른 테이블의 PK를 참조하여 테이블 간의 연결 고리를 만드는 키입니다. 참조 무결성을 보장하여 존재하지 않는 회원이 대여를 하거나, 없는 책을 대여하는 오류를 막아줍니다.

- 1:N 연결 방식: '1'에 해당하는 부모 테이블의 PK를, 'N'에 해당하는 자식 테이블이 FK 컬럼으로 가짐으로써 데이터를 유기적으로 연결합니다.

**Q3. SQL의 핵심 구문들(SELECT/INSERT/UPDATE/DELETE)은 각각 언제 쓰이나요?**
- SELECT: 저장된 데이터를 조회, 검색, 정렬, 집계하여 필요한 정보를 뽑아낼 때 사용합니다.

- INSERT: 테이블에 새로운 행(데이터)을 추가할 때 사용합니다.

- UPDATE: 이미 존재하는 행의 특정 컬럼 값을 수정할 때 사용합니다. (예: 반납 시 대여 상태 변경)

- DELETE: 테이블에서 더 이상 필요 없는 행을 삭제할 때 사용합니다.

**Q4. JOIN과 GROUP BY로 "연결된 데이터를 한 번에 뽑는 방법"은 무엇인가요?**
- 핵심 메커니즘: 관계형 데이터베이스에서 데이터는 여러 테이블에 쪼개져 저장됩니다. JOIN은 이 분산된 테이블들을 FK(외래키) 관계를 기준 삼아 옆으로 이어 붙여 하나의 커다란 가상 테이블로 만드는 역할을 합니다. 이렇게 연결된 상태에서 GROUP BY를 적용하면, 특정 기준(예: 회원별, 카테고리별)으로 데이터를 그룹핑하여 단 한 줄의 쿼리로 원하는 집계 데이터를 뽑아낼 수 있습니다.

- 실무적 흐름:

    1. FROM member m INNER JOIN rental r ON m.id = r.member_id 로 회원 정보와 대여 기록을 먼저 매칭(연결)합니다.

    2. GROUP BY m.id 를 통해 동일한 회원의 대여 기록들을 하나의 그룹으로 묶습니다.

    3. SELECT m.name, COUNT(r.id) 로 각 그룹(회원)에 속한 대여 건수를 집계합니다.

- 의의: 이 구조를 이해하면 여러 번 DB를 조회할 필요 없이, 단 한 번의 쿼리로 '어떤 회원이 책을 몇 권 빌렸는지'와 같은 복합적인 요구사항을 효율적으로 해결할 수 있습니다.

**Q5. 실무에서 흔한 요구(검색/정렬/집계/랭킹)를 SQL로 어떻게 해결하나요?**
실무 비즈니스 요구사항은 SQL의 핵심 기능들과 1:1로 매칭하여 직관적으로 풀어나갈 수 있습니다.

- 검색 (Filtering):

    - WHERE 절과 LIKE 연산자를 사용합니다.

    - 예: 특정 키워드가 포함된 도서 검색 (WHERE title LIKE '%파이썬%')

- 정렬 (Sorting):

    - ORDER BY 절을 사용하며, 최신순이나 높은 가격순 등 기준 컬럼과 정렬 방향(ASC/DESC)을 지정합니다. 필요에 따라 LIMIT과 조합하여 상위 N개만 끊어옵니다.

    - 예: 최근에 대여된 기록 상위 5개 조회 (ORDER BY rented_at DESC LIMIT 5)

- 집계 (Aggregation):

    - GROUP BY와 집계 함수(COUNT, SUM, AVG, MAX, MIN)를 조합합니다. 조건에 맞는 집계 그룹을 필터링할 때는 HAVING 절을 사용합니다.

    - 예: 카테고리별 평균 도서 가격 (GROUP BY category_id + AVG(price))

- 랭킹 (Ranking):

    - 표준 SQL의 윈도우 함수(Window Function)인 ROW_NUMBER(), RANK(), DENSE_RANK() 등을 사용하거나, 집계된 데이터를 ORDER BY ... DESC로 정렬하여 순위를 맥입니다.

    - 예: 가장 대여가 많이 된 도서 순위 매기기 (COUNT(id) 기준 정렬)

**Q6. 인덱스(INDEX)가 왜 필요하며, 어떤 컬럼에 적용하면 좋은가요?**
- 필요성: 책의 '찾아보기(색인)'와 같은 역할을 합니다. 데이터의 양이 많아질 때 테이블 전체를 처음부터 끝까지 읽지 않고(Full Scan), 원하는 데이터를 빠르게 찾기 위해 필요합니다.

- 적용 대상: WHERE 절에서 검색 조건으로 자주 쓰이는 컬럼, JOIN의 연결 고리가 되는 컬럼, 또는 ORDER BY로 자주 정렬되는 컬럼에 적용하면 조회 성능이 크게 향상됩니다.
